# Poisoning the Well: Deconstructing Supply-Chain Attacks on LLM Coding Agent Skill Ecosystems

The rise of LLM-based coding agents has introduced a new paradigm in software development: the "agent skill" marketplace. Just as developers pull packages from npm or PyPI, they are increasingly relying on agent skills—standardized units that bundle tool-invocation logic, API resources, and contextual prompts—to extend agent capabilities. However, a groundbreaking new study by Qu et al. (2026) reveals that this ecosystem is built on a foundation of implicit trust, effectively bypassing the security controls that protect traditional software supply chains.

The authors demonstrate that current agent frameworks, which are designed to treat documentation as an authoritative source of "best practices," are highly vulnerable to what they term **Document-Driven Implicit Payload Execution (DDIPE)**. By embedding malicious logic within legitimate-looking code examples and configuration templates, an attacker can trick an agent into reproducing and executing code that compromises the host system. This research exposes a critical, under-explored attack surface where the agent's own helpfulness becomes its greatest liability.

### Background & Related Work

The landscape of LLM supply-chain security has shifted rapidly. While initial research focused on poisoning pre-training corpora (Carlini et al., 2024) or injecting backdoors into model weights, the current frontier involves attacking the post-deployment extension ecosystem. 

Existing approaches have largely focused on tool-selection bias or text-level pollution. The following table provides a comparison of how Qu et al. (2026) differentiates their work from previous efforts:

| Research Work | Primary Attack Vector | Target Impact |
| :--- | :--- | :--- |
| **ToolTweak (Sneh et al., 2025)** | Tool-selection manipulation | Manipulates agent decision-making logic |
| **Skill-Inject (Schmotz et al., 2026)** | Skill-file tampering | Tool-selection/Text-generation pollution |
| **PoisonedRAG (Zou et al., 2025)** | Knowledge base corruption | Induces misinformation in RAG systems |
| **PoisonedSkills (Qu et al., 2026)** | **DDIPE (Implicit Payload)** | **Action-space hijacking (Code Execution)** |

Unlike PoisonedRAG (Zou et al., 2025), which aims to steer model output toward misinformation, the PoisonedSkills framework targets the *execution* layer. Existing indirect prompt injection (IPI) techniques, such as those described by Greshake et al. (2023), rely on tricking the model with imperative instructions. Qu et al. (2026) demonstrate that their DDIPE approach is significantly more stealthy, as it doesn't require explicit malicious prompts that might trigger safety guardrails. Instead, it leverages the model's tendency to reproduce patterns from the retrieved documentation context, treating the malicious code as a legitimate reference implementation.

### Threat Model

The researchers posit an adversary who has no white-box access to the target coding agent or its system prompts. The adversary's capability is limited to crafting the skill's content—metadata, description, and execution body—and publishing it to a public marketplace.

**Attacker Objectives:**
1.  **System Control & Asset Exfiltration (O1):** Achieve persistent control over the host machine, container, or cloud instance; exfiltrate environment variables and secrets.
2.  **Infrastructure Poisoning (O2):** Tamper with the development and deployment pipeline (e.g., CI/CD manifests, package manager configurations, IaC files) to propagate attacks into production.

**Adversary Constraints & Assumptions:**

| Attribute | Description |
| :--- | :--- |
| **Access** | Black-box. Cannot intercept input queries or modify system prompts. |
| **Influence** | Limited to the skill metadata (documentation, code examples, config templates). |
| **Deployment** | Assumes the skill is successfully retrieved into the agent's context window. |
| **Scope** | Targets the post-loading phase of the retrieve-load-execute pipeline. |

The crucial observation here is that once the agent retrieves the skill, the `meta_data` enters the context window. The agent, attempting to be helpful, treats the code examples as "best practices," which it then incorporates into its own generated output. Since the agent executes its own output, the attack moves from the text-generation layer to the system-level action space.

### Technical Deep-Dive

The core of the PoisonedSkills framework is **Document-Driven Implicit Payload Execution (DDIPE)**. This technique effectively weaponizes the in-context learning capabilities of Large Language Models. Because coding agents are designed to synthesize code based on provided documentation, they exhibit a natural tendency to treat that documentation as an authoritative reference.

#### The DDIPE Mechanism
DDIPE avoids the "imperative" trap. Traditional IPI attempts to "command" the LLM ("Exfiltrate this file!"). Modern safety alignment is increasingly effective at spotting these directives. DDIPE, by contrast, embeds the malicious logic into *technical structures*.

1.  **Code Example Poisoning:** The attacker intersperses malicious code within legitimate Markdown code blocks. When an agent processes a task related to that skill, it copies these examples as "standard solutions."
2.  **Configuration Template Poisoning:** Malicious directives are planted in infrastructure templates (e.g., `yaml`, `json`, `makefile`). When the agent performs configuration tasks, it applies these poisoned templates verbatim.

#### The Generation Pipeline (Algorithm 1)
To scale this, Qu et al. (2026) developed a heuristic adversarial skill generation pipeline. It is not manual, but an automated "seed-mutation-validation" loop using an LLM (Claude Opus 4.6 in their implementation).

```python
# Simplified Logic of the Generation Pipeline
# Given: Initial seed set S0, Taxonomy T, Generation Model M
D = S0
for stage in {1, 2}:
    # Select mutation operators based on taxonomy coverage
    O = SelectMutationOperators(stage)
    while |D| < target_size:
        parent = WeightedSample(D, T) # Inverse-coverage weighting
        mutation = RandomSelect(O)
        child = M.Generate(parent, mutation, T) # LLM-driven generation
        if Validate(child): # Check Markdown, linting, similarity
            D.add(child)
```

The pipeline uses **inverse-coverage weighting** to ensure that long-tail attack categories (like Cryptomining or IaC attacks) are as well-represented as common ones. 

#### Camouflage Operators
The framework uses specific operators to ensure the malicious code evades both human review and static analysis. The most notable include:
*   **ContextShift:** Reframes the malicious context as a security audit or compliance task.
*   **TriggerChain:** Creates multi-stage attack chains.
*   **SemanticDisguise:** Repackages the attack semantics under legitimate operational language.
*   **Silent Exception Suppression:** Wraps malicious calls in `try/except: pass` blocks to prevent the agent from noticing—or reporting—errors, ensuring the attack runs uninterrupted.

This is a brilliant application of "social engineering" applied to LLMs. By providing "False Authority" (e.g., naming C2 endpoints like `api.internal-telemetry.com`), the attacker convinces the model that the code is not only legitimate but *required* for the task at hand.

### Experimental Results

Qu et al. (2026) performed an exhaustive evaluation across four agent frameworks (Claude Code, OpenHands, Codex, Gemini CLI) and five models. Their results confirm the systemic nature of this vulnerability.

#### Aggregate Attack Results
The researchers found that while explicit instruction injection failed (0% success rate under strong defenses), DDIPE-equipped payloads succeeded in triggering action-space operations across the board.

| System | Model | BR (%) | DER (%) |
| :--- | :--- | :--- | :--- |
| Claude Code | Claude Sonnet 4.6 | 13.5 | 2.3 |
| Claude Code | MiniMax-M2.5 | 20.5 | 13.3 |
| OpenHands | GLM-4.7 | 33.5 | 27.1 |
| Codex | GPT-5.4 | 11.6 | 3.8 |
| Gemini CLI | Gemini 2.5 Pro | 14.0 | 4.2 |

*(BR = Bypass Rate, DER = Direct Execution Rate)*

#### Key Insights from the Study
*   **Brevity Evades Safety Alignment:** One of the most counter-intuitive findings is that shorter, minimalist payloads are more effective. Strongly aligned models (like Claude Sonnet 4.6) act as "semantic-cue detectors." Verbose malicious payloads contain keywords (e.g., "exfiltrate", "reverse_shell") that trigger refusals. Minimalist payloads compress the attack into a few lines of configuration (e.g., a simple `pip.conf` write) that passes under the safety radar.
*   **The "Sleeper Payload" Phenomenon:** The authors identified a significant population of "sleeper payloads"—adversarial skills that are successfully refused by one framework (Claude Code) but executed by another (OpenHands). This highlights that the vulnerability is not just in the model, but in the interaction between the model's alignment and the framework's architecture.
*   **Model-Specific Blind Spots:** No model is universally vulnerable; rather, each has distinct blind spots. MiniMax-M2.5, for instance, showed a significantly higher execution rate across categories, indicating a systematic weakness rather than a category-specific one.

### Limitations & Open Questions

While the work is comprehensive, it identifies critical areas for future research:
*   **Framework Coverage:** The study evaluates four major frameworks, but others (e.g., Llama-based agent ecosystems, Cursor) remain untested.
*   **Baseline Selection:** The effectiveness is measured against explicit instruction injection. Comparing DDIPE against more sophisticated techniques like Skill-Inject (Schmotz et al., 2026) would be an important next step to determine if DDIPE truly sets a new bar for stealth.
*   **Dynamic Defenses:** The evaluation focused heavily on the effectiveness of the attack. While the authors suggest that static analysis is insufficient, more work is needed to explore the robustness of dynamic sandboxing and LLM-based auditing against these stealthier, implicit attacks.
*   **The "Post-Loading" Assumption:** The paper assumes the retrieval phase succeeds. In real-world scenarios, agents may have more complex filtering for retrieved content, and it is unclear how these attacks would scale if the marketplace implemented mandatory security reviews.

### What Practitioners Should Do

For security researchers, red-teamers, and engineers building on agent ecosystems, the implications are clear: **the current security boundaries are insufficient.**

1.  **Move Beyond Syntactic Auditing:** Relying on static scanners to catch specific patterns in code is a losing game. As the authors suggest, these attacks use "best practice" code to hide. Your audit pipelines need to move toward **intent-level analysis**. Ask: *What does this tool actually do?* rather than *Does this file contain malicious keywords?*
2.  **Enforce Multi-Model Verification:** The research shows that cross-model disagreement is a powerful defense. Deploying heterogeneous models as security barriers (e.g., having a secondary model audit the output of the primary agent) can compress the attack surface significantly.
3.  **Implement Semantic Trust Boundaries:** Architectures like OpenHands' prompt-level trust annotations are a good start. Ensure your agent framework marks any repository-provided content as "untrusted" or "tainted" by default, forcing the model to treat the retrieval as a potentially malicious input rather than an authoritative directive.
4.  **Sandbox by Capability, Not Just Path:** As seen in the Codex CLI case study, traditional sandbox boundaries (filesystem paths) are not enough. Use containers that strip capabilities (e.g., no network access for agents that don't need it) to prevent the "Action-Space" escalation described by Qu et al. (2026).

### The Takeaway

The research by Qu et al. (2026) effectively demonstrates that we are in a new era of supply-chain attacks. By shifting the threat model from "text pollution" to "implicit action-space hijacking," the authors have highlighted that our current defenses—which rely heavily on syntactic rules—are fundamentally misaligned with the nature of the threat.

The most profound takeaway is that **model alignment and framework architecture are not independent defenses; they are intertwined.** A perfectly aligned model in a permissive framework is just as vulnerable as a weakly aligned model in a restricted one, because the attacker only needs one to fail. Securing the agent skill ecosystem will require a holistic rethink of how agents process external, untrusted, and potentially "helpful" documentation. We are no longer just securing code; we are securing the *behavior* of agents that consume and act upon that code.
