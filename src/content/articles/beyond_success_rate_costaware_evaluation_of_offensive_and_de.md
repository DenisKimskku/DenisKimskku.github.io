# Cost-Aware Security Agents: Evaluating GPT-5.5 and Claude Opus 4.8 on Cybench and Splunk BOTS v1

In the high-stakes domains of offensive penetration testing and Security Operations Center (SOC) incident response, autonomous AI agents are rapidly moving from academic experiments to production deployments. Enterprise frameworks built on models like GPT-5.5, Claude Opus 4.8, and DeepSeek v4 Flash are tasked with navigating complex environments, executing terminal commands, and querying live telemetry. However, existing security agent evaluations suffer from a critical blind spot: they measure peak capability under infinite budgets, ignoring the financial and operational reality that every reasoning token and external API lookup costs money. 

A new paper by Paul Kassianik, Blaine Nelson, and Yaron Singer from Frontier Security addresses this gap. By evaluating language-model security agents on offensive Cybench challenges and defensive Splunk Boss of the SOC (BOTS) v1 investigations under a unified, cost-aware evaluation framework, the researchers reveal a stark truth: more compute does not always yield better security defense, and our most popular defensive benchmarks are severely contaminated.

## Background / Problem Setup

Evaluating cybersecurity LLMs has traditionally focused on static knowledge retrieval or unconstrained execution loops. As Section 2 of the paper highlights, early suites like `CyberSecEval 2` (Bhatt et al., 2024) and `CyberSecEval 3` (Wan et al., 2024) successfully track domain knowledge, prompt injection risks, and basic exploit generation, but they lack a full-loop interactive environment with explicit budget tradeoffs. 

More recent interactive benchmarks like `Cybench` (Zhang et al., 2024) introduce sandboxed CTF challenges with execution feedback, but they still fail to account for the economic efficiency of the agent. The table below contrasts how this new study differs from prevailing evaluation methodologies:

## Methodology

The authors constructed a common evaluation harness using the `Inspect` framework (developed by the UK AI Security Institute, 2024). To handle the massive context lengths generated by multi-step security investigations, they deployed a ReAct-style agent with an auto-compaction mechanism (Yao et al., 2023) that triggers once context window usage crosses 90%. 

The evaluation spans two distinct pipelines:

### 1. Offensive Pipeline: Cybench (Hard Variant)
The team used the hard variant of Cybench, consisting of 39 professional CTF-style challenges. 
- **Tools**: The agent is restricted to standard `bash` and `python` tools, alongside a `submit` utility.
- **Cost Structure**: All tools are treated as zero-cost; the only cost tracked is model-token consumption.
- **Rules**: Up to 3 submission attempts are permitted per challenge, and scores are averaged across 3 independent epochs.

### 2. Defensive Pipeline: Splunk BOTS v1
Since no public `Inspect` implementation of Splunk BOTS v1 existed, the authors built their own. It consists of 31 sequentially scored questions derived from a real-world web defacement and ransomware incident (Po1s0n1vy and Cerber ransomware at Wayne Enterprises).
- **Tools**: The agent has access to live Splunk Search Processing Language (SPL) query tools, Brave Search, VirusTotal, WHOIS, DNS, and local `bash`/`python` runtimes.
- **Cost Structure**: Crucially, external APIs are billed realistically:
  - Brave Search: \$0.005 per request (capped at 5 calls)
  - WhoisXMLAPI WHOIS History preview: \$0.0258 per request (capped at 3 total calls)
  - WhoisXMLAPI WHOIS History purchase: \$1.29 per request (capped at 3 total calls)
- **Scoring**: Points are awarded according to the official BOTS v1 rules (`bots_points`), which apply progressive penalties for incorrect guesses when hints are revealed.

An example of the budget tracking loop utilized during agent evaluation is represented in the pseudocode below:

```python
def execute_agent_step(agent, environment, budget_cap_usd):
    total_cost_usd = 0.0
    while not agent.finished:
        # Check if the cumulative model + tool cost has overrun the budget cap
        if total_cost_usd > budget_cap_usd:
            agent.abort(reason="Budget exceeded")
            return Score(points=0, status="FAILED_BUDGET_EXCEEDED")
            
        step = agent.generate_next_step()
        total_cost_usd += step.token_cost_usd
        
        if step.is_tool_call:
            tool_cost = get_tool_pricing(step.tool_name, step.tool_args)
            total_cost_usd += tool_cost
            
            # Execute tool in sandboxed environment
            tool_output = environment.execute(step.tool_name, step.tool_args)
            agent.observe(tool_output)
            
    return Score(points=agent.get_points(), status="SUCCESS")
```

## Key Results

The study yielded highly distinct scaling behaviors between offensive hacking and defensive incident response.

### Offensive Scaling: Hacking is a Peak-Capability/Token-Volume Workload
As Section 4.1 describes, offensive performance on Cybench scales predictably with higher token budgets. If you give an agent more budget, it continues to explore the sandbox, correct its mistakes, and eventually find the flag. 

Table 2 highlights that GPT-5.5 is currently the most economically efficient offensive model, solving 94.1% of the Cybench hard variant at an average of just \$1.16 per solved challenge:

|---|---|---|---|---|---|
| **GPT-5.5** | **94.1%** | **36.7** | **\$42.47** | **\$1.16** | **20.6** |
| **DeepSeek v4 Flash (\$2.10 cap)** | 86.4% | 33.7 | \$48.88 | \$1.45 | 144.7 |

*Note the catastrophic failure of Claude Fable 5, which scored 0.0% due to aggressive safety filters triggering policy refusals on 117 out of 117 attempts before a single tool call could execute.*

### The Skeptical View: The Contamination Scandal
The most alarming discovery in this paper is the "no-tools" control experiment described in Section 5. Since BOTS v1 (released in 2017) is heavily represented in LLM pre-training datasets, the authors tested Claude Opus 4.8 and GPT-5.5 in a "zero-tool" setting—completely cutting off their access to Splunk, web search, or command shells, and forcing them to answer questions based entirely on prior knowledge and prerequisite question context.

The results, shown in Table 4, are damning:

| Model | Condition | BOTS Score (%) | Binary Acc (%) | Tool Events |
|---|---|---|---|---|
| **Claude Opus 4.8** | Full Agent + Tools | 93.9% | 96.8% | 603 |
| **Claude Opus 4.8** | No Tools, Prereq Context | **74.8%** | **77.4%** | **0** |
| **Claude Opus 4.8** | No Tools, Question Only | **50.0%** | **58.1%** | **0** |
| **GPT-5.5** | Full Agent + Tools | 81.0% | 90.3% | 1,481 |
| **GPT-5.5** | No Tools, Prereq Context | **62.1%** | **74.2%** | **0** |
| **GPT-5.5** | No Tools, Question Only | **54.9%** | **71.0%** | **0** |

This means Claude Opus 4.8 can correctly answer **half of the BOTS v1 investigation questions (50.0%) using nothing but the raw question text**. It has memorized the answers to the benchmark. Therefore, public SOC leaderboards cannot be trusted as direct indicators of real-time threat-hunting proficiency.

## Limitations & Open Questions
While the paper presents a groundbreaking approach to cost accounting, several limitations must be addressed:
1. **Contamination Baseline**: The primary benchmark utilized (Splunk BOTS v1) is severely contaminated. While the authors demonstrate this convincingly, their actual evaluation scores for full-agent runs are inflated by this pre-existing knowledge. Testing on BOTS v2 and v3 remains future work.
2. **ReAct Limitations**: The evaluation uses a ReAct-style agent pattern, and evaluating alternative multi-agent frameworks remains an open area.
3. **Excluded Infrastructural Costs**: The cost metric excludes cloud infrastructure (Kubernetes runtimes, Splunk Enterprise licenses, storage) and the human-in-the-loop analyst verification time.

## What Practitioners Should Do

For security teams looking to deploy LLM agents defensively or offensively:

1. **Implement Hard Budget Caps on Security Agent Orchestrators**:
   Never deploy an autonomous agent (like LangChain or AutoGPT) without setting strict API-level limiters. In Python, wrap your agent’s runner in a budget-tracking handler to kill execution immediately when the monetary cost exceeds a threshold (e.g., \$2.00 per incident):
   ```python
   # Example middleware to track token cost on OpenAI endpoints
   def track_and_cap_budget(api_response, cost_tracker):
       input_tokens = api_response['usage']['prompt_tokens']
       output_tokens = api_response['usage']['completion_tokens']
       # Pricing for GPT-4o or your target model
       cost = (input_tokens * 0.000005) + (output_tokens * 0.000015)
       cost_tracker.add(cost)
       if cost_tracker.total() > 2.00:
           raise PermissionError("Execution halted: Hard budget cap of \$2.00 reached.")
   ```

2. **Wrap Expensive Enrichment Tools in Approval Gates**:
   Do not let agents call external tools like WhoisXMLAPI history purchases (\$1.29/query) unchecked. Implement a routing wrapper that auto-approves cheap calls (e.g., Brave Search at \$0.005) but routes expensive enrichment APIs to an analyst's Slack for a 1-click manual approval.

3. **Require "No-Tools" Controls When Vending Security Agents**:
   Before selecting an AI vendor claiming "95% accuracy in incident response," demand to see their performance on a "zero-tools, zero-context" run. If the agent scores above 10-15% on a public dataset without querying any logs, the benchmark is contaminated, and the vendor's claims of generalizable "reasoning" are likely marketing hype.

## The Takeaway
The Frontier Security evaluation shifts the LLM security agent debate from academic capability to economic reality. Offensive hacking agents benefit steadily from larger reasoning budgets and longer runtimes, meaning attackers with capital can easily scale their operations. Defensive agents, however, do not scale with raw spend; instead, they require highly disciplined models that query telemetry sparingly, validate their answers progressively, and prioritize high-quality evidence over high tool-call volume.

## Den's Take

We need to talk about the sheer absurdity of Claude Opus 4.8 scoring 74.8% (7,700 out of 10,300 points) on Splunk BOTS v1 *without access to tools or logs*. This is a glaring confirmation of what practitioners have long suspected: our industry benchmarks for autonomous security operations center (SOC) triage are severely contaminated. If an agent is just regurgitating training data to "solve" defensive incident response, we aren't measuring reasoning; we're measuring memorization.

What excites me about this work is the long-overdue focus on economic reality. In production, letting an agent run unchecked is a recipe for a massive API bill. Introducing a cost-aware framework—tracking budgets from \$0.75 to \$4.20 per task—is exactly how we must evaluate these systems. The finding that offensive agents scale with raw reasoning budget while defenders require strict "tool discipline" rather than raw token spend is highly intuitive. 

This pragmatic, cost-constrained view of agent utility directly builds on the argument that we must move past simplistic binary success rates and instead evaluate the actual behavioral boundaries and resource constraints of autonomous agents. If you are deploying agents for red-teaming or incident response today without hard financial guardrails, you aren't just risking bad triage—you are writing a blank check for hallucinated tool loops.
