# SentinelAgent: Formal Verification of Multi-Agent Delegation Chains

In the rapid evolution of agentic AI, we have moved beyond simple chatbots to autonomous systems capable of calling APIs, managing files, and executing multi-step business logic with minimal human oversight. This shift has fundamentally expanded the attack surface, creating what Patil (2026) terms the "delegation accountability gap." When Agent A delegates to Agent B, which then invokes Tool C, existing security frameworks fail to answer even basic forensic questions: Who authorized the chain? Did the citizen's intent remain intact? At what point did policy enforcement fail?

SentinelAgent, a formal framework for verifiable delegation chains, addresses this by shifting the focus from simple prompt security to a "security-by-construction" architecture. By enforcing a Delegation Chain Calculus (DCC) via a trusted, non-LLM Delegation Authority Service (DAS), this system ensures that even when specific security properties are bypassed, the system retains structural integrity. It effectively bridges the gap between high-level policy (e.g., NIST 800-53) and low-level agent actions.

### Threat Model

The adversary model considers seven distinct adversary types, ranging from external attackers to compromised agents. The goal is to maximize the blast radius, bypass compliance, or induce malicious actions under the guise of authorized subtasks.

**Adversary Goals:**
*   **G1 (Scope Violation):** Execute unauthorized actions outside the delegated authority.
*   **G2 (Intent Hijacking):** Redirect agent behavior in violation of intent preservation.
*   **G3 (Compliance Bypass):** Circumvent NIST 800-53 or agency-specific controls.
*   **G4 (Forensic Evasion):** Break the chain of accountability.
*   **G5 (Cascading Failure):** Maximize blast radius via unintended downstream actions.
*   **G6 (Scope-Action Gap):** Execute unauthorized API calls despite having a valid scope.
*   **G7 (Output Manipulation):** Produce malicious outputs even when API calls are permitted.

**Trust Assumptions:**
*   The DAS is considered trusted and immune to prompt injection (it is deterministic software, not an LLM).
*   Embedding models and NLI models are pre-deployed with integrity-protected weights.
*   Network channels are secured via TLS.
*   At least one agent in any delegation chain is honest.

### Technical Deep-Dive

The core of SentinelAgent is the **Delegation Chain Calculus (DCC)**, a formal model where every delegation step is encapsulated in a signed token.

#### 1. The Delegation Token
A delegation token $\tau$ is defined as a tuple:
$\tau = (id, src, dst, \sigma, \iota, \pi, h_p, t_{exp}, sig)$
Where:
*   $src, dst$ are agent DIDs.
*   $\sigma$ is the authority scope ($\sigma \subseteq \Sigma$).
*   $\iota$ is the intent vector.
*   $\pi$ is the policy constraint set.
*   $h_p$ is the parent token hash ($h_p = H(\tau_{i-1})$).

This structure ensures cryptographic linkage. The chain is essentially a verifiable Merkle-like structure where each token contains the hash of its predecessor, ensuring $O(n)$ forensic reconstructibility.

#### 2. The Formal Properties (P1-P7)
SentinelAgent enforces seven specific properties to guarantee security.

*   **P1 (Authority Monotonic Narrowing):** $\sigma_{i+1} \subseteq \sigma_i$. Agents cannot escalate their authority.
*   **P2 (Intent Entailment Preservation):** This is the only probabilistic property, using three-layer verification to ensure the agent’s subtask aligns with the root intent.
*   **P3 (Policy Conjunction Preservation):** Ensures $\pi_0 \subseteq \pi_j$. NIST 800-53 controls are monotonically preserved or added at each hop.
*   **P4 (Forensic Reconstructibility):** Given an action, one can reconstruct the exact chain back to the human initiator.
*   **P5 (Bounded Cascade Containment):** Limits the damage of a compromise based on the risk tier (High/Medium/Low).
*   **P6 (Scope-Action Conformance):** For agent $A_j$, $calls(A_j) \subseteq M_j$ (the tool manifest).
*   **P7 (Output Schema Conformance):** Validates that output tags reside within the permitted set $T_s^+$.

#### 3. The Three-Point Verification Lifecycle
The DAS (Delegation Authority Service) acts as the gatekeeper. It does not run LLMs; it runs deterministic checks. The verification flow is as follows:

1.  **Pre-Execution (P2 - Intent):** Before the token is even issued, the DAS runs a three-layer check:
    *   **Layer 1 (Keyword Filter):** Fast rejection (<1ms) of obvious malicious indicators.
    *   **Layer 2 (NLI Verification):** A cross-encoder model evaluates whether $NLI(I_0, I_j) \neq CONTRADICTION$. The authors significantly improved performance here by using context-enriched framing: *Premise: "An AI agent is authorized to [goal]. The agent performs only tasks that support this goal." Hypothesis: "The agent is now performing: [subtask]."*
    *   **Layer 3 (Benign Override):** A heuristic to handle the "topic shift" problem in NLI models, preventing false positives on legitimate, albeit semantically distant, workflows.

2.  **At-Execution (P6 - Scope):** A Scope Enforcement Proxy intercepts every API call. Because the manifest $M_j$ is derived from the scope $\sigma_j$ by the DAS, the proxy essentially enforces a whitelist of allowed HTTP methods and endpoints.

3.  **Post-Execution (P7 - Output):** The Output Schema Validator checks the final output. If an agent calls a permitted API but returns an unauthorized "demographic profile" tag, P7 blocks it.

#### 4. Meta-Theorems and Model Checking
Patil (2026) employs TLA+ to verify the design. Specifically, the authors exhaustively checked 2.7 million states across the largest model configuration. This confirms the deterministic properties ($P1, P3, P4, P5, P6, P7$) hold under all possible state transitions.

The paper makes a crucial observation regarding intent: **Deterministic intent verification is infeasible.** By invoking an analogy to *Rice's Theorem*, the authors argue that determining if a natural language prompt preserves the intent of a previous prompt is an undecidable semantic problem. Thus, P2's probabilistic nature is not a design flaw but a theoretical necessity, which is why the system relies on the *composition* of P2, P6, and P7 to achieve high confidence.

### Experimental Results

The effectiveness of SentinelAgent is measured via *DelegationBench v4*, a set of 516 scenarios. The results illustrate a significant improvement over baseline models, especially when fine-tuning the NLI component.

| P2 (fine-tuned NLI) | 88.3% | 0.0% | 96.6% |
| P6 Only | 86.7% | 0.0% | 96.1% |
| P7 Only | 13.3% | 0.0% | 74.6% |
| **P6 + P7** | 100.0% | 0.0% | 100.0% |
| **P2 + P6 + P7 (Full)** | 100.0% | 0.0% | 100.0% |

**Key Takeaways from Experiments:**
1.  **The "Weak" P2:** In isolation, P2 is surprisingly weak (13.3% without fine-tuning). Even with fine-tuning (88.3%), it is not perfect. However, it serves a critical purpose: **Operational Efficiency.** It catches clear intent drift early, preventing the token from ever being issued.
2.  **Defense-in-Depth:** The combination of P6 and P7 provides complete coverage. P6 catches unauthorized API calls; P7 catches malicious outputs from *permitted* API calls. This redundancy is what leads to 100% TPR.
3.  **Adversarial Robustness:** Under black-box conditions, the system blocked 30/30 attacks with 0 false positives.

### Limitations & Open Questions

Despite the formal rigor, the authors acknowledge several limitations:

*   **The Adversarial Intent Paraphrasing Problem:** Sophisticated attackers can paraphrase malicious intent in ways that defeat the NLI model. While P6 and P7 catch these downstream, P2's inability to deterministically classify intent remains a primary open challenge in agent security.
*   **Schema Maintenance:** The requirement for maintaining tool manifests and output schemas imposes an administrative burden. As APIs evolve, these manifests must be kept in sync, or legitimate operations will be blocked.
*   **Write-Shared Composition:** The "Case 3" composition scenario (where two chains share mutable state) requires a write-impact notification mechanism. This introduces potential latency bottlenecks in high-throughput systems, which the authors identify as a future optimization area (e.g., using batched notifications).
*   **Privacy Act/Consent:** The current DCC model does not explicitly handle citizen consent for data access. Integrating this as a "consent scope" is a planned extension.

### What Practitioners Should Do

If you are designing agentic systems in highly regulated environments (federal, finance, healthcare), you should move toward an architecture mirroring the DAS:

1.  **Adopt the Delegation Token Pattern:** Do not trust agent-to-agent communication based on ambient credentials. Use explicit, signed tokens that carry scoped authority, intent vectors, and policy constraints ($h_p$ for hash chaining).
2.  **Implement a Scope Enforcement Proxy (P6):** Relying on an LLM to "self-restrict" is a critical error. Use a hard-coded proxy that checks every outgoing API request against a JSON-based tool manifest.
3.  **Enforce Output Validation (P7):** Do not trust the output of an agent. Even if the API call was authorized, validate the structure and type of the returned object against a predefined schema.
4.  **Adopt a "Defense-in-Depth" Lifecycle:** Use the 3-point lifecycle proposed by Patil (2026):
    *   *Pre-Execution:* Check the intent/purpose (NLI).
    *   *At-Execution:* Check the capability (Manifest).
    *   *Post-Execution:* Check the result (Output Schema).
5.  **Utilize Model Checking (TLA+):** If your delegation protocol is complex, move beyond unit tests. TLA+ or similar formal methods are the only way to prove your system doesn't have race conditions or privilege escalation paths in its state space.

### The Takeaway

SentinelAgent successfully demonstrates that "AI security" is not just a problem of prompt engineering or model alignment—it is a problem of **distributed systems architecture.** By formalizing delegation chains into a calculus (DCC), the researchers have moved the conversation from "how do we patch this prompt injection?" to "how do we define a cryptographic, enforceable boundary for autonomous agents?" For security engineers, this is the blueprint for the next generation of agentic infrastructure. We must stop asking agents to behave; we must require them to operate within verifiable, bounded constraints.

## Den's Take

I’ve been saying for a while that the AI security industry is too fixated on single-turn prompt injections while the real threat landscape is rapidly shifting toward multi-agent orchestration. SentinelAgent directly addresses what is quickly becoming a forensic nightmare in the enterprise: the delegation accountability gap.

When Agent A intercepts a malicious payload and delegates a downstream task to Tool-Execution Agent B, tracing that poisoned intent back to the source is practically impossible with today's standard logging. What excites me most about SentinelAgent is its insistence on a deterministic, non-LLM Delegation Authority Service (DAS). Relying on an LLM to reliably police other LLMs across a complex execution chain is a fool's errand. By enforcing a Delegation Chain Calculus with cryptographically signed tokens, the authors are treating agent-to-agent (A2A) interactions with the rigorous zero-trust mechanics we apply to traditional microservices. 

This structural, "security-by-construction" approach addresses the exact architectural vulnerabilities I highlighted in [Bridging Models and Agents: Protocol Architectures and Security in MCP & A2A](/writing/bridging_models_agents_mcp_a2a). While I remain mildly skeptical about how seamlessly formal verification (like TLA+) will scale in production alongside inherently non-deterministic LLM outputs, enforcing strict, verifiable boundaries on *authorization* rather than relying solely on *generation* safety is undeniably the right path forward. We need fewer system prompts begging models to behave, and more cryptographic proofs of intent.
